The Partner Shell, Navigation, Role Access, and Cashier Sub-RoleBefore we build any individual feature, we need the container that holds everything together. Right now the partner (currently called "client" in the system) logs in and sees some existing pages like dashboard, catalog, onboarding, checklist. We need to restructure this into a clear three-phase navigation that matches the store lifecycle, and we need to support two types of people inside a partner's store — the owner and the cashier.When a store owner logs in with the "partner" role, their top navigation should organize everything into clear sections that represent the three phases of their journey. Phase A is "Store Setup" — this is where they plan and build their physical store. Phase B is "Products & Orders" — this is where they browse our catalog, place orders, pay, and track deliveries. Phase C is "Store Operations" — this is the POS, inventory, reports, and daily management stuff we already defined in the POS prompts. There should also be a "My Account" section for profile, settings, and support.The key insight here is that these phases aren't just tabs — they represent a journey that unlocks progressively. When a partner first signs up and completes onboarding, they should land in Phase A because their store doesn't exist yet. Phase B (catalog and ordering) should be available from the start because they need to browse and understand what products they'll stock. Phase C (POS and daily ops) should be locked until their store status is flipped to "active" by our ops team — this is the same locking mechanism we defined in the POS prompts. The navigation should visually show which phase they're in — maybe the Phase C tabs appear grayed out with a small lock icon and a tooltip saying "Available when your store is live."Now for the cashier sub-role. The store owner should be able to add staff members from their settings page. When they add a cashier, that person gets a separate login but with extremely limited access — they can ONLY see the billing screen, the stock receive page, and the held bills. They cannot see: the product catalog and ordering (Phase B), the store setup (Phase A), inventory reports, daily sales reports, cash register history, store settings, payment milestones, invoices, or any financial data. The cashier sees one thing: the billing screen ready to go, with a small menu to access stock receive and their active held bills. That's it. Think of it like Marg's "Cashier" role vs "Manager" role — the cashier operates the register, the owner runs the business.For now, implement the cashier management with mock data — a JSON array of staff members with name, phone, role (owner or cashier), pin code (a 4-digit number they use to identify themselves at the register), and active/inactive status. The store owner sees a "Staff" section in settings where they can add, edit, and deactivate cashiers. When the POS system connects to a real database later, each sale will record which cashier made it — this is important for the cash register reconciliation and for the owner to see performance per cashier.The existing client pages (dashboard, onboarding, checklist, catalog, profile, support, etc.) should be reorganized into this new structure but not rebuilt — just moved into the right phase section. The dashboard becomes the landing page and should adapt based on where the partner is in their journey: if they're in the setup phase, the dashboard highlights setup progress and checklist completion. If their store is active, the dashboard shows today's sales, stock health, and recent orders — essentially a summary that pulls from Phase B and Phase C data.All data for the partner portal in this prompt should use mock JSON files that can be easily swapped for real API calls later. Create a folder of mock data files: mockStaff.json, mockStoreConfig.json (store name, address, GSTIN, settings), and mockStoreStatus.json (status: "setup" or "active", package: "Lite"/"Pro"/"Elite", onboarding step, launch date). Every component should import from these mocks through a single data service layer so the swap to real APIs is a one-file change per entity.
